## 描述

現狀：`classify.py` 在拆分任務時，通常只針對單一的 Idea 檔案。
優化：在呼叫 LLM 進行分類前，先餵給它目前的 `PROJECTS.md` 摘要。
* 好處：防止 LLM 建立名稱極度接近的新專案（例如已有 gold-analysis 卻又建立 gold-monitor），讓它能更精準地建議將任務歸類到「現有專案」中。

## 實作細節

1. 修改 `/Users/claw/skills/clw-ideas2tasks/scripts/classify.py` 中的 `classify_idea` 函數
2. 在呼叫 `detect_category` 和 `build_tasks` 之前，先讀取 `~/Tasks/PROJECTS.md` 並提取專案摘要
3. 將這個摘要作為額外的上下文傳遞給分類過程
4. 修改 `detect_category` 函數以接受額外的上下文參數
5. 在上下文中，我們可以提供：
   - 現有專案名稱列表
   - 每個專案的簡短描述（從 README.md 提取前幾行）
   - 最近更新的專案
6. 更新分類提示詞，讓 LLM 知道要優先考慮現有專案
7. 為了避免破壞現有功能，如果讀取 PROJECTS.md 失敗，則回退到原行為

## 需求

1. 在 `classify.py` 中添加新函數 `get_projects_context()`：
   - 讀取 `~/Tasks/PROJECTS.md`
   - 解析專案列表表格（從 `## 📋 專案列表` 開始）
   - 提取每個專案的名稱、狀態、進度、最後更新日期
   - 格式化為適合作為 LLM 上下文的字符串
   - 返回上下文字符串，如果失敗則返回空字符串

2. 修改 `detect_category(text: str, context: str = "")`：
   - 添加可選的 `context` 參數
   - 如果有上下文，則在關鍵詞匹配時考慮上下文中的專案名稱
   - 例如，如果上下文中提到「gold-analysis」，則對於包含「gold」或「黃金」的文本，提升該專案的匹配分數

3. 修改 `classify_idea(idea: dict) -> dict`：
   - 在函數開頭調用 `get_projects_context()` 獲取上下文
   - 將上下文傳遞給 `detect_category` 和可能的其他函數
   - 在返回的結果中，可以考慮添加一個字段表示是否使用了上下文

4. 需要修改的文件：
   - `/Users/claw/skills/clw-ideas2tasks/scripts/classify.py`

## 產出

- 修改後的 `classify.py` 包含上下文感知功能
- 在 `ideas2tasks-improvements` 項目的 README 中更新進度
- 可選：在 `PROJECTS.md` 中顯示此改進的狀態

## 驗收標準

- 當 `PROJECTS.md` 中已存在類似「gold-analysis」專案時，輸入一個想法「監控黃金價格波動並通知」時，LLM 建議的專案名稱應該是「gold-analysis」或「gold-monitor」等現有專案，而不是創建一個全新的專案如「gold-tracker」。
- 當想法完全不匹配任何現有專案時，LLM 仍能提議建立新專案。
- 讀取 `PROJECTS.md` 失敗時，系統仍能正常運行（降級處理）。
- 修改不應該影響現有的分類準確性，只應該在有明確匹配時偏向現有專案。
