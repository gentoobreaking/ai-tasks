## 描述

* 配置管理：目前依賴系統環境變數，可以考慮在 Skill 目錄下支援 `.env` 檔案或 `config.yaml`。
* 單元測試：目前的 Frontmatter 解析和狀態標準化使用了大量的正則表達式，建議針對 `state_sync.py` 撰寫單元測試 (Unit Test)，確保未來修改邏輯時不會導致 290 多個任務解析失敗。

## 需求

### 配置管理
1. 在 `ideas2tasks` 技能目錄下（例如 `~/.opencode/skill/ideas2tasks/`）支援一個配置檔案，如 `.env` 或 `config.yaml`。
2. 定義配置項目，例如：
   - `GITHUB_REPO`: 預設的 GitHub 倉庫
   - `IDEAS_DIR`: Ideas 目錄的絕對路徑
   - `TASKS_DIR`: 任務目錄的絕對路徑
   - 其他目前透過環境變數或硬編碼的設定。
3. 修改腳本（如 `executor.py`, `classify.py` 等）在啟動時先讀取此配置檔案，然後再讀取環境變數（環境變數覆蓋配置檔案），最後使用硬編碼預設值。
4. 提供一個範例配置檔案（如 `config.yaml.example` 或 `.env.example`）在技能目錄中。
5. 確保不會破壞現有依賴環境變數的設置方式。

### 單元測試
1. 選擇一個核心腳本，例如 `state_sync.py`（或類似負責 Frontmatter 解析和狀態同步的腳本）。
2. 為此腳本建立單元測試檔案，例如 `tests/test_state_sync.py`。
3. 使用標準的 Python 單元測試框架（`unittest` 或 `pytest`，視專案現有習慣而定）。
4. 測試內容應該包括：
   - 正確解析各種格式的 Frontmatter（包括有無、有多餘空白、不同欄位順序）。
   - 正確更新 Frontmatter（例如將 status 從 pending 改為 done）。
   - 邊界情況：缺失欄位、無效日期格式等。
   - （如果適用）狀態同步函式的行為。
5. 確保測試可以在本地機器上運行（不需要真實的 GitHub 倉庫）。
6. 在 README 中說明如何執行測試。

## 產出

- 配置管理：修改後的腳本以支援配置檔案，以及範例配置檔案。
- 單元測試：新增的測試檔案和（可選）測試執行指令。
- 在 `ideas2tasks-improvements` 項目的 README 中更新進度。
- 可選：在 `PROJECTS.md` 中顯示此改進的狀態。

## 驗收標準

### 配置管理
- 當在技能目錄下放置一個 `.env` 檔案並設定 `IDEAS_DIR=/tmp/test_ideas` 時，腳本會使用此目錄作為 Ideas 目錄（而不是預設或環境變數）。
- 環境變數仍然可以覆蓋配置檔案的設定。
- 當兩者都未設定時，使用硬編碼預設值（或現有行為）。

### 單元測試
- 建立至少一個單元測試檔案，並且所有測試通過。
- 測試覆蓋 Frontmatter 解析的主要情況。
- 未來對該腳本的修改不會導致現有功能中斷（透過測試可以快速驗證）。
